# AI 狼人杀 MVP 设计说明

日期：2026-05-05

## 1. 目标

将现有 React 单机前端原型改造成一个可用的 MVP，支持：

- 单人玩家进入房间并开始一局游戏
- AI 补齐其余玩家，完整跑通一局狼人杀
- 游戏使用 4 个基础角色：狼人、预言家、女巫、平民
- 前后端通过 WebSocket 通信
- 后端使用 Python 3 + LangChain
- AI 玩家模型使用硅基流动的 Qwen3 8B

本阶段的优先级是：

1. 可完整游玩
2. 规则和流程稳定
3. AI 发言有基本沉浸感

## 2. MVP 范围

### 包含

- 完整产品流：房间/准备页 -> 游戏中 -> 结算页
- 后端作为唯一真实状态源
- 前端通过 WebSocket 接收状态和事件
- 白天讨论阶段 AI 生成短发言
- 玩家在白天讨论阶段可输入一小段短发言
- 夜晚行动、投票、胜负判断由后端规则/策略驱动
- 基础断线处理至少做到：前端重连后可重新拉起当前局面展示（如实现成本可控）

### 不包含

- 多人真人同房
- 自由实时聊天系统
- 所有 AI 完全 agent 化自治
- 守卫、猎人等扩展角色
- 观战、复盘、排行榜、持久存档

## 3. 总体架构

### 前端 React

职责：

- 展示房间页、游戏页、结算页
- 维护本地 UI 状态（连接状态、输入框、按钮可用性、当前展示数据）
- 建立 WebSocket 连接并收发协议消息
- 渲染后端下发的游戏快照和事件流

不负责：

- 真实规则判定
- 阶段推进
- 胜负判断
- AI 行为计算

### 后端 Python 3 + WebSocket

职责：

- 维护唯一真实游戏状态
- 创建和管理单局 session
- 角色分配
- 阶段推进
- 夜晚行动结算
- 投票结算
- 胜负判断
- 玩家动作合法性校验
- 构造玩家可见视图
- 推送事件和状态快照

### AI 层（LangChain + Qwen3 8B）

职责拆分为两层：

1. 策略层（非 LLM）
   - 夜晚行动决策
   - 投票目标决策
   - 白天发言意图生成

2. 生成层（LLM）
   - 将角色、人设、当前局面摘要、发言意图转换为 1~3 句自然语言短发言

约束：

- 夜晚决策和投票优先使用规则/策略，不依赖模型自由决定
- 模型主要负责讨论文本生成
- LangChain 主要承担 prompt 组织、上下文拼接、模型调用封装和后续可观测性扩展

## 4. 产品流程

### 4.1 房间/准备页

展示内容：

- 玩家昵称输入
- 本局玩法说明
- 固定角色集合说明
- 开始按钮

行为：

- 前端连接后端并创建 session
- 玩家输入昵称后点击开始
- 后端初始化游戏并分配角色
- 前端进入身份揭示/游戏页

### 4.2 游戏页

页面区域建议：

- 顶部：当前阶段、回合数、连接状态
- 左/右侧：玩家列表（仅显示公共可见信息）
- 中间：事件与发言时间线
- 底部：当前操作区

操作区根据阶段动态变化，例如：

- 等待中
- 选择查验对象
- 选择是否救人
- 选择毒人目标
- 白天输入短发言
- 投票对象选择
- 点击继续

### 4.3 结算页

展示内容：

- 胜负结果
- 全员身份公开
- 关键事件摘要
- 再来一局按钮

## 5. 后端核心模块划分

### session_manager

职责：

- 管理单局 session 生命周期
- 维护 websocket 连接
- 保存 session_id 与当前游戏实例映射

### game_engine

职责：

- 定义 GameState
- 处理角色分配
- 推进阶段
- 结算夜晚行动
- 结算投票
- 判断胜负
- 校验玩家动作是否合法

要求：

- 纯规则核心尽量与传输层解耦
- 可独立单测

### visibility_builder

职责：

- 从完整 GameState 构造发给前端的人类玩家视图
- 控制私有信息与公共信息隔离

示例：

- 玩家可见自己的身份
- 其他 AI 只展示公共信息
- 结算时才展示所有角色

### ai_strategy

职责：

- 计算 AI 夜晚动作
- 计算 AI 投票目标
- 输出 AI 白天发言意图

### ai_dialogue

职责：

- 接收 AI 发言意图与上下文
- 调用 Qwen3 8B 生成短发言
- 做长度与内容约束

### message_schema

职责：

- 定义前后端 websocket 消息结构
- 统一事件类型和负载字段

## 6. 数据模型

后端完整状态建议至少包含：

```text
GameState
- session_id
- phase
- round
- status
- players
- public_log
- private_log
- night_actions
- vote_state
- winner
```

### Player

```text
Player
- id
- name
- is_human
- role
- alive
- persona
- known_info
```

### 前端视图模型

```text
GameView
- session_id
- phase
- round
- self_player
- player_list
- timeline
- available_actions
- winner
- connection_state
```

原则：

- 前端只收到当前玩家该看到的内容
- 不直接下发其他 AI 的真实身份和私密推理

## 7. 阶段流转设计

MVP 主链路：

1. 房间准备
2. 身份揭示
3. 夜晚开始
4. 狼人行动
5. 预言家查验
6. 女巫决策
7. 夜晚结算
8. 白天公告
9. 白天讨论
10. 玩家投票
11. AI 投票
12. 投票结算
13. 胜负判断
14. 下一轮或游戏结束

设计约束：

- 后端驱动节奏推进
- 当前阶段如需玩家操作，后端发 `action_required`
- 当前阶段如为 AI 操作，后端内部完成后推送事件
- 前端不负责判断“下一步该到哪个阶段”

## 8. 白天讨论与玩家发言

### AI 发言

- 每个 AI 在白天讨论阶段发言一次
- 单次发言限制为 1~3 句
- 发言按固定顺序进行
- AI 发言由“策略意图 + LLM 生成”两层构成

### 玩家发言

- 玩家在白天讨论阶段可输入一小段短发言
- 后端将该发言写入公共时间线
- 后续 AI 发言可将玩家发言作为上下文之一

限制：

- 不做自由实时聊天
- 不做多轮抢话与打断
- 仍由后端控制整体发言顺序和节奏

## 9. WebSocket 协议建议

### 前端 -> 后端

- `create_session`
- `start_game`
- `submit_action`
- `submit_discussion_message`
- `request_next`
- `ping`

### 后端 -> 前端

- `session_created`
- `state_snapshot`
- `game_event`
- `action_required`
- `action_result`
- `game_over`
- `error`

### 设计原则

- 状态类消息与事件类消息分离
- `state_snapshot` 用于同步当前可见状态
- `game_event` 用于构建时间线体验
- `action_required` 明确告知前端当前用户可执行动作

## 10. 前端重构方向

当前项目中，前端已经内置了大量游戏规则和流程逻辑，例如：

- `src/components/WerewolfGame.jsx`
- `src/game/gameEngine.js`
- `src/ai/aiEngine.js`

MVP 重构方向：

- 前端页面和组件保留可复用 UI 部分
- 将规则状态机从前端迁出到后端
- 将 `WerewolfGame.jsx` 从“本地规则驱动容器”改为“WebSocket 视图容器”
- 前端保留展示层和交互层，移除关键规则判定职责

## 11. 测试边界

### 规则测试

至少覆盖：

- 角色分配正确
- 夜晚行动结算正确
- 女巫解药/毒药逻辑正确
- 投票结算正确
- 胜负判断正确

### 协议测试

至少覆盖：

- 正确阶段下发正确 `action_required`
- 非法 action 被拒绝
- 状态快照不会泄露不该暴露的身份信息

### 体验验证

至少覆盖：

- 能从房间页进入游戏并完整打到结算页
- AI 发言可正常展示
- 玩家白天短发言可进入时间线并影响后续 AI 上下文
- WebSocket 异常时前端能给出基本状态提示

## 12. MVP 之后的自然演进

如果 MVP 跑通，下一阶段可以顺着当前架构继续扩展：

- 增加守卫、猎人等角色
- 支持多人房间
- 增加复盘与关键决策回顾
- 强化 AI 人设与长期记忆
- 增加重连恢复与持久化

## 13. 推荐实现原则

- 先做后端唯一状态源，再重接前端页面
- 先保证规则正确和协议清晰，再增强 AI 表现
- 前端优先做稳定信息架构，不在首版追求复杂动画
- AI 文本生成受控、短句、可预测，避免模型主导规则

## 14. 结论

本 MVP 推荐采用“前后端职责清晰版”：

- React 前端负责产品流和交互展示
- Python 后端负责规则、状态、节奏与校验
- LangChain + Qwen3 8B 负责讨论期 AI 发言生成
- 规则决策尽量走确定性策略而不是完全交给模型

该方案最适合当前目标：在可控复杂度内，把现有单机原型升级为一个真正可玩的 AI 狼人杀 MVP。
